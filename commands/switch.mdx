---
title: "npm run switch — rotate Kiro IDE accounts via SSO cache"
description: "Switch the active Kiro IDE account from a saved template, with optional auto-restart and Windows machine ID rotation to appear as a new device."
---

The `switch` command reads a Builder ID template file produced by `npm run register` and writes the next account's credentials into the AWS SSO cache that Kiro IDE reads on startup. It tracks which account was used last so subsequent invocations rotate through the list automatically. Optionally it can restart Kiro immediately after switching and, on Windows, randomise the `MachineGuid` registry value so the machine appears to be a new device.

## Usage

```bash
npm run switch [-- [flags]]
```

Pass flags after `--` to separate them from the npm invocation. When called with no arguments the command opens the interactive menu.

## Flags

<ParamField query="--template" type="string" default="show/builderid-template.json">
  Path to the Builder ID template file generated by `npm run register`. Must be a JSON array where each entry has at minimum `refreshToken`, `clientId`, and `clientSecret`.
</ParamField>

<ParamField query="--region" type="string" default="us-east-1">
  AWS region written into the token cache file (`~/.aws/sso/cache/kiro-auth-token.json`).
</ParamField>

<ParamField query="--startUrl" type="string" default="https://view.awsapps.com/start">
  SSO start URL. Used to compute the SHA-1 hash that names the client registration cache file. Change this only if your Kiro installation uses a non-default start URL.
</ParamField>

<ParamField query="--state" type="string" default="show/switch-state.json">
  Path to the state file that tracks which account index will be used next. The file is created automatically on the first switch.
</ParamField>

<ParamField query="--dry-run" type="boolean" default="false">
  Simulate the switch without writing any files. The selected account and all computed values are printed to stdout, but `~/.aws/sso/cache/` and the state file are left unchanged. Useful for verifying the template before committing to a switch.
</ParamField>

<ParamField query="--index" type="number">
  Pick a specific zero-based account index instead of using round-robin. The value is clamped to the range `[0, accounts.length - 1]`. When omitted the next index from the state file is used.
</ParamField>

<ParamField query="--restart" type="boolean" default="false">
  Automatically stop the running Kiro process and restart it after the account switch completes. Alias: `-r`. Without this flag a reminder to restart Kiro manually is printed at the end.
</ParamField>

<ParamField query="--reset-machine-id" type="boolean" default="false">
  Generate a new random UUID and write it to the Windows `MachineGuid` registry key after switching accounts, making the machine appear as a new device. Alias: `-m`.

  <Warning>This flag only works on Windows and requires the process to be running with administrator privileges. On other platforms it exits silently. See the [machine ID reset](#machine-id-reset) section for details.</Warning>
</ParamField>

<ParamField query="--interactive" type="boolean" default="false">
  Force interactive menu mode even when other flags are present. Aliases: `-i`. Interactive mode is also the default when no arguments are supplied.
</ParamField>

## Interactive menu

When launched without arguments (or with `--interactive`), the command shows a menu that displays live status: account count, next account index, Kiro process state, and the current machine ID (Windows only).

| Key | Action |
|-----|--------|
| `1` | Switch to the next account (manual Kiro restart required) |
| `2` | Switch account and restart Kiro automatically |
| `3` | Switch account, reset machine ID, and restart Kiro |
| `4` | Show detailed status: template path, account count, next index, Kiro PID, machine ID |
| `5` | Kill the running Kiro process |
| `6` | Start Kiro |
| `7` | Restart Kiro without switching accounts |
| `8` | Reset the Windows MachineGuid only (no account switch) |
| `0` | Exit |

## Examples

<CodeGroup>

```bash Interactive mode (default)
npm run switch
```

```bash Switch account and restart Kiro
npm run switch -- --restart
```

```bash Full reset: switch + machine ID + restart
npm run switch -- --restart --reset-machine-id
```

```bash Dry run to preview without changes
npm run switch -- --dry-run
```

```bash Switch to a specific account index
npm run switch -- --index 2 --restart
```

```bash Use a custom template file
npm run switch -- --template ./accounts/my-template.json --restart
```

</CodeGroup>

## Files written by switch

### `~/.aws/sso/cache/kiro-auth-token.json`

The primary token file read by Kiro on startup. Written on every non-dry-run switch.

```json
{
  "accessToken": "",
  "refreshToken": "aor1...",
  "expiresAt": "2025-01-01T00:00:00.000Z",
  "clientIdHash": "<sha1-of-startUrl>",
  "authMethod": "IdC",
  "provider": "BuilderId",
  "region": "us-east-1"
}
```

### `~/.aws/sso/cache/<clientIdHash>.json`

Client registration file. The filename is the SHA-1 hash of `{"startUrl":"<startUrl>"}`.

```json
{
  "clientId": "...",
  "clientSecret": "...",
  "expiresAt": "2025-04-01T00:00:00.000",
  "scopes": [
    "codewhisperer:completions",
    "codewhisperer:analysis",
    "codewhisperer:conversations",
    "codewhisperer:transformations",
    "codewhisperer:taskassist"
  ]
}
```

### `show/switch-state.json`

Tracks the next account index for round-robin rotation. Created automatically on the first switch.

```json
{
  "nextIndex": 1
}
```

<Note>Before overwriting any existing cache file, the switch command creates a timestamped backup next to the original (e.g. `kiro-auth-token.json.bak.1700000000000`). You can restore a previous account by renaming the backup file.</Note>

## Machine ID reset

The `--reset-machine-id` / `-m` flag writes a new random UUID to the Windows registry key:

```
HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Cryptography\MachineGuid
```

<Warning>Administrator privileges are required to modify this registry key. Right-click your terminal and choose "Run as administrator" before running `npm run switch -- --reset-machine-id`. If the process does not have sufficient privileges the operation fails with an access-denied error, but the account switch itself is unaffected.</Warning>

This feature is a no-op on macOS and Linux; a warning is printed and execution continues normally.

## Workflow

<Steps>
  <Step title="Register accounts">
    Run `npm run register` to create one or more AWS Builder ID accounts and generate `show/builderid-template.json`.
  </Step>
  <Step title="Switch to the next account">
    Run `npm run switch -- --restart` to write the next account's credentials into the AWS SSO cache and restart Kiro.
  </Step>
  <Step title="Repeat as needed">
    Each invocation advances the round-robin index. Run `npm run switch` again whenever you need to rotate to the next account.
  </Step>
</Steps>
